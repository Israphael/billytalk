# Решение: выбор стека

**Статус:** принято
**Дата:** 2026-07-19

## Решение

**Python 3.14 + wxPython, с разделением на headless-ядро и отдельный процесс интерфейса,
связанные локальным IPC.**

```
┌──────────────────────────┐        ┌─────────────────────────┐
│  ЯДРО (headless)         │        │  ИНТЕРФЕЙС (wxPython)   │
│  ─────────────────       │  IPC   │  ────────────────────   │
│  хук ввода               │ ◄────► │  трей                   │
│  захват микрофона        │ JSON-  │  окно настроек          │
│  провайдеры расшифровки  │  RPC   │  история                │
│  вставка текста          │        │  плашка записи          │
│  хранилище истории       │        │                         │
└──────────────────────────┘        └─────────────────────────┘
   нет UI-поверхности                 заменяем, не трогая ядро
```

## Почему не Tauri — главный вывод разведки

Естественный выбор для такого продукта — Tauri. Он **исключён**, и причина не в
производительности.

**NVDA трактует любое окно WebView2 как веб-страницу и проваливается в browse mode.**
Пользователь вынужден жать NVDA+Space перед каждым вводом. Из [NVDA #19655]:

> «NVDA обрабатывает окно приложения как обычную веб-страницу, по умолчанию переходя в
> режим просмотра… Пользователи вынуждены многократно вручную переключать режим фокуса,
> что серьёзно нарушает поток работы.»

Лечится это только тем, что кто-то в NV Access напишет app-module персонально под наше
приложение — так сделано для VS Code и Skype. Нативные UIA-приложения такого разговора
не ведут вообще.

Дополнительно:

- **У Tauri нет AccessKit.** Проверено: в `Cargo.toml` у `tao` и `wry` на ветке `dev` и в
  дереве `platform_impl` под Windows — ноль зависимостей на AccessKit. Вся доступность
  отдана чужому рантайму Edge. Всё, что вне webview (меню трея, нативные диалоги),
  получает только умолчания Win32.
- **Рантайм не наш.** WebView2 — это Edge *пользователя*. Открытый с декабря 2025
  [NVDA #19329]: области `aria-live` не озвучиваются в WebView2, при том что
  «в Edge и Chrome этот пример работал». Для диктовки `aria-live` — **это и есть** канал
  статуса: «слушаю», «расшифровываю», «вставил». Обойти закреплением версии нельзя.
- **Реакция мейнтейнеров на баги доступности слабая.** [tauri#15624] завёл дизайнер
  доступности из GitHub, протестировав 19 сборок против NVDA, JAWS и Voice Access.
  Тикет едва не закрыли как специфичный для GitHub. Потребовался инженер Microsoft по
  доступности WebView2 и минимальное воспроизведение на голых `tao`+`wry`, прежде чем
  мейнтейнер признал: «Значит, это всё-таки баг на стороне Tauri». Исправление
  ([PR #15625]) до сих пор не влито.

**Выбрав Tauri сегодня, мы убиваем версию 3.0.** Это самое дешёвое решение принять
правильно сейчас и самое дорогое — переигрывать потом.

Если веб-интерфейс когда-нибудь окажется необходим — брать **Electron**, не Tauri:
там Chromium наш, доступность на уровне Chrome, и разрыва с `aria-live` нет.

## Почему wxPython

**Решающий факт: NVDA сам написан на wxPython.**

В исходнике `source/gui/__init__.py` — `import wx`, `import wx.adv`,
`class SysTrayIcon(wx.adv.TaskBarIcon)`. Самый распространённый в мире бесплатный
скринридер, сделанный незрячими для незрячих, — это wx-приложение с wx-треем.

Отсюда два следствия:

1. **wx оборачивает нативные контролы Win32**, поэтому доступность наследуется, а не
   реализуется. Мы получаем её бесплатно и заранее.
2. **Диалог назначения хоткеев списываем с NVDA буквально** — тулкит тот же.
   Их `gui/inputGestures.py` решает ровно нашу задачу: как дать человеку назначить
   сочетание клавиш, когда скринридер сам перехватывает эти сочетания. Их модель —
   явный модальный режим захвата, в котором скринридер намеренно отступает.

### Рассмотренные альтернативы

| Тулкит | Вердикт |
|---|---|
| **wxPython** | ✅ выбран |
| Win32 / WinForms | эталон доступности, но чужой язык для нашего ядра |
| Qt (PySide6) | настоящий мост UIA, рабочий вариант, но хвост багов на стороне NVDA |
| WinUI 3 | ❌ самый новый стек Microsoft оказался наименее доступным |
| tkinter | ❌ поддержки скринридеров не было до Tk 9.1; в CPython её подключают прямо сейчас |
| Flutter desktop | ❌ символы в текстовых полях не читаются при перемещении стрелками — смертельно для диктовки |
| Tauri | ❌ см. выше |

## Почему Python

**Режим №3 живёт в Python.** Локальная расшифровка с выбором модели и анализом железа —
это faster-whisper / ctranslate2, где Python родной. На Rust пришлось бы дублировать
экосистему ради того же результата.

**Проекты-конкуренты ушли в Rust по причине, которой у нас нет.** Handy, Whispering,
openless — все на Tauri+Rust, но все они делают локальный инференс как основной режим:
это упирается в вычисления и требует соседства с нативными ML-рантаймами. У нас
основной режим — тонкий HTTP-клиент. Единственный проект с нашей архитектурой
(WhisperWriter) написан на Python.

**Скорость сборки.** MVP планируется за один-два цикла. Python даёт наибольший объём
рабочего кода за цикл.

## Почему ядро отделено от интерфейса

Стоит один день. Покупает:

- **Ядро не имеет UI-поверхности** — значит не может испортить доступность.
- **Интерфейс заменяем** без единого касания логики. Версия 3.0 может получить
  собственную поверхность, не ломая версию 1.0.
- **Ядро тестируется без графики** — вся логика диктовки покрывается тестами напрямую.
- **Хук ввода живёт на отдельном потоке с собственным насосом сообщений**, как того
  требует Windows, и не конкурирует с циклом событий интерфейса.

Последнее — не стилистика, а требование выживания. Если обработчик низкоуровневого хука
превысит `LowLevelHooksTimeout` (по умолчанию 1000 мс), **Windows молча снимает хук и
никак об этом не сообщает**. Диктовка перестаёт работать посреди сессии, и приложение об
этом не узнает. Обработчик обязан только положить событие в очередь и вернуться.

## Сопутствующие решения

| Вопрос | Решение | Причина |
|---|---|---|
| Установщик | NSIS, режим `currentUser` | Ставится без UAC. Автозапуск всё равно пользовательский: установщик с правами администратора пишет автозапуск в чужой профиль. |
| Автозапуск | `HKCU\...\Run`, запись при установке | Единственный механизм, видимый и отключаемый в Параметрах Windows. Планировщик задач там невидим. |
| Права | Обычные, без повышения | Повышенный процесс **не сможет вставлять текст** в обычные приложения — то есть сломает основную функцию ради редкого случая. |
| Настройка хоткея | В приложении при первом запуске, не в установщике | Стандартный контрол `msctls_hotkey32` не умеет кнопки мыши в принципе. Плюс тихая установка страницу установщика пропускает. |
| Ключ API | Диспетчер учётных данных Windows | Не файл и не реестр. |
| Обновления | Встроенный механизм с подписью minisign | Поддерживает заголовки авторизации — это готовый крючок под платный канал в будущем. |
| Упаковка | PyInstaller 6.21 | Nuitka на 3.14 по собственному журналу изменений «прошла отбор лишь едва». |
| Подпись кода | Пока не подписываем | EV больше не снимает предупреждение SmartScreen — это подтверждает сама Microsoft. К платному тарифу вернуться к вопросу. |

## Что это решение сознательно отдаёт

Честно, чтобы потом не удивляться:

- **Размер дистрибутива** больше, чем у Tauri (~8 МБ). Python + wxPython + рантайм дадут
  порядка 40–60 МБ. Приемлемо.
- **Ложные срабатывания антивирусов** у PyInstaller хуже, чем у нативных сборок:
  самораспаковывающийся архив — ровно та форма, на которую реагируют эвристики.
- **Время старта** требует внимания. Окно интерфейса при старте создавать нельзя —
  запускаемся только в трей, окно поднимаем по требованию. Иначе Windows пометит
  приложение как «высокое влияние на запуск».

Ни одно из трёх не затрагивает основную функцию и не блокирует дорожную карту.

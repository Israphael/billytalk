# Решение: хук остаётся в Python, нативный помощник не нужен

**Статус:** принято, подтверждено измерением
**Дата:** 2026-07-19
**Отменяет:** предложение вынести хуки в отдельный нативный процесс

## Предыстория

Обзор под линзой платформы утверждал, что чистый Python-хук на ctypes нежизнеспособен:

> «Колбэк срабатывает на каждое движение мыши. DeathAdder V2 X HyperSpeed — мышь
> с высокой частотой опроса, до 1000 колбэков в секунду, каждый с захватом GIL…
> постоянная точка захвата GIL в системном пути ввода.»

Предлагалось вынести оба хука в нативный процесс на C или Rust (~200 строк).
Рассуждение выглядело убедительным, поэтому решение было предварительно принято.

**Измерение его опровергло.**

## Измерение

Реальное железо заказчика, 90 секунд активного движения мыши, Python 3.14.6,
`WH_MOUSE_LL` через ctypes, колбэк только считает события.

```
события за 90 с   : 11 639   (11 547 движений, 92 кнопки и колесо)
средняя частота   : 129 событий/с
пик (окно 5 с)    : ~460 событий/с
колбэк среднее    : 2,5 мкс
колбэк p50        : 1,7 мкс
колбэк p99        : 14,6 мкс
колбэк МАКСИМУМ   : 497 мкс
процессорное время: 0,03% одного ядра
```

**Худший колбэк — 497 мкс при бюджете `LowLevelHooksTimeout` в 1 000 000 мкс.
Запас в 2000 раз.**

Частота оказалась на порядок ниже предсказанной: 129 в секунду вместо 1000. Частота
опроса мыши и частота событий низкоуровневого хука — разные величины; система
не отдаёт хуку каждый аппаратный отчёт.

Скрипт: `spikes/s0c_hook_getmsg.py`, сырой вывод: `spikes/s0c-run.log`.

## Решение

**Хуки остаются в Python.** Экономия: целый нативный компонент, нативный тулчейн
в сборке, третья граница IPC и сопровождение кода на двух языках.

## Обязательные меры

Запас большой, но хвост распределения измерением не покрыт — сборка мусора по
большой куче и подгрузка модуля под нагрузкой не воспроизводились. Поэтому:

1. **Колбэк только кладёт событие в очередь.** Никакой работы внутри.
2. **`gc.freeze()` после инициализации** — переводит стартовые объекты в постоянное
   поколение и убирает их из обходов сборщика.
3. **PyInstaller в режиме `--onedir`, не `--onefile`.** Onefile распаковывает весь
   пакет во временную папку при каждом запуске — это и десятки мегабайт ввода-вывода
   на старте (ломает требование по влиянию на запуск из спецификации), и подгрузка
   модулей с удержанием GIL уже во время работы хука.
4. **Сторож остаётся.** Тихое снятие хука системой обнаружить изнутри нельзя,
   поэтому периодическая проверка живости и переустановка нужны независимо от запаса
   по времени.

## Ловушка, найденная тем же спайком

**Цикл `PeekMessage` со сном не получает колбэки низкоуровневого хука вообще.**

Первая версия замера крутила:

```python
while ...:
    while u.PeekMessageW(byref(msg), None, 0, 0, PM_REMOVE):
        ...
    time.sleep(0.002)
```

Результат: **ноль событий за 30 секунд активного движения мыши.** При этом
`SetWindowsHookExW` вернул валидный дескриптор, `GetLastError` — ноль. Всё выглядит
работающим, событий нет.

Работает только **блокирующий `GetMessage`**:

```python
while u.GetMessageW(byref(msg), None, 0, 0) > 0:
    u.TranslateMessage(byref(msg))
    u.DispatchMessageW(byref(msg))
```

Выход из цикла — `PostThreadMessageW(tid, WM_QUIT, 0, 0)` с другого потока.

**Это обязательное требование к реализации.** Отладка «хук поставился, событий нет»
без этого знания стоит часов: ошибок нет, дескриптор валиден, документация умалчивает.

## Урок к процессу

Правдоподобное техническое рассуждение от авторитетного источника оказалось неверным
в главном численном утверждении. Час измерения отменил архитектурное решение, которое
добавило бы в проект целый нативный компонент.

# Результаты аппаратной пробы ввода

**Дата:** 2026-07-19
**Железо:** Razer DeathAdder V2 X HyperSpeed, VID_1532 / PID_009C, беспроводной донгл
**ОС:** Windows 11 Pro 10.0.26200
**Инструмент:** `C:\BillyTalk\probes\billytalk_mouse_probe.py` (голый ctypes, без зависимостей)

## Итог: лучший сценарий, рисков не выявлено

Боковые кнопки отдают стандартные Windows-события XBUTTON. Специальная обработка
vendor-HID не требуется.

```
Mouse 4 →  WM_XBUTTONDOWN / WM_XBUTTONUP   mouseData=0x00010000 → HIWORD=1 → XBUTTON1
Mouse 5 →  WM_XBUTTONDOWN / WM_XBUTTONUP   mouseData=0x00020000 → HIWORD=2 → XBUTTON2
```

## Проверенные факты

| Проверка | Результат | Следствие для проекта |
|---|---|---|
| Стандартные XBUTTON | ✅ да | Достаточно `WH_MOUSE_LL`; Raw Input не нужен |
| Флаг `LLMHF_INJECTED` | ❌ **ноль событий** | Synapse не перехватывает; события от железа |
| Razer Synapse запущен | ❌ нет процессов и служб | Работает встроенная прошивочная раскладка |
| Press **и** release | ✅ оба | Удержание (push-to-talk) реализуемо |
| Замер длительности | ✅ тап 180 мс / удержание 1079 мс | Порог тап-vs-удержание 200–250 мс разделяет надёжно |
| Клавиатурный хук параллельно | ✅ работает | Запасной клавиатурный хоткей без отдельного механизма |
| Кнопки 6+ | отсутствуют на этой модели | Проблема недостижимости кнопок 6+ неактуальна |

## Извлечение номера кнопки — место, где все ошибаются

Номер X-кнопки лежит в **старшем слове** `mouseData`, а не в самом `wParam`:

```c
WORD xbtn = HIWORD(ms->mouseData);   // (mouseData >> 16) & 0xFFFF
// XBUTTON1 = 1 → «Mouse 4» (по умолчанию «Назад»)
// XBUTTON2 = 2 → «Mouse 5» (по умолчанию «Вперёд»)
```

Младшее слово зарезервировано. Маскирование всего DWORD против 0x1/0x2 молча не работает.

## Аномалия: Raw Input не доставил событий

`RegisterRawInputDevices` с `RIDEV_INPUTSINK` в message-only окно вернул успех,
но ни одного события `WM_INPUT` не пришло. Причина не установлена.

**Влияние на проект: нулевое** — низкоуровневый хук закрывает задачу целиком, Raw Input
рассматривался только как запасной канал. Зафиксировано, чтобы не тратить время на
повторное расследование.

## Составное устройство: 10 коллекций

Донгл HyperSpeed выставляет устройство как составное:

- 2 × MOUSE (`MI_00`, `MI_01&Col02`)
- 2 × KEYBOARD (`MI_02`, `MI_01&Col01`)
- 6 × vendor-HID

**Практическое следствие.** Сейчас Mouse 4 приходит по мышиному каналу. Но если
пользователь откроет Synapse и переназначит кнопку на макрос, событие уедет в
клавиатурную коллекцию и мышиный хук её потеряет — кнопка «перестанет работать»
без единого сообщения об ошибке.

**Требование к реализации:** слушать оба канала одновременно и при пропаже мышиных
событий от привязанной кнопки показывать пользователю внятную подсказку про Synapse,
а не молчать.

## Риск, который не реализовался

Разведка выделяла отдельный поток на «проблему Razer»: Synapse умеет подменять боковые
кнопки на клавиатурные макросы на уровне драйвера или прошивки, и тогда стандартный хук
слепнет. AutoHotkey и Wispr Flow независимо пришли к одному обходному пути — просить
пользователя самому назначить кнопке клавиатурное сочетание в фирменном софте.

**На этом железе в текущей конфигурации проблемы нет.** Обходной путь всё равно стоит
задокументировать в справке — у других пользователей Synapse будет запущен.

## Гигиена

Пробник по устройству логирует все нажатия клавиш. Файл `probe-run.log` содержал
клавиатурный ввод за время сеанса и подлежит удалению после извлечения строк мыши.

Это же соображение переносится в архитектуру продукта: буфер аудио и текст расшифровки
должны быть типами **без сериализатора и без строкового представления**, чтобы попадание
их в лог было ошибкой компиляции, а не пунктом чеклиста.

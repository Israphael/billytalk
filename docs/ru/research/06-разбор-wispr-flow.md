# Разведка: разбор установленного Wispr Flow

**Дата:** 2026-07-18/19
**Версия:** Wispr Flow 1.6.122
**Метод:** чтение установки на машине заказчика — только на чтение, ничего не менялось.

## Границы

Читались структура файлов, схема настроек, схема базы данных и логи. **Не читались:**
содержимое расшифровок, значения токенов, зашифрованные DPAPI данные. Схема — да,
содержимое — нет.

## Зачем это делалось

Заказчик пользуется Wispr Flow и назвал две проблемы: обрыв диктовки и вставка не в то
поле. Разбор нашёл причины обеих — это превращает жалобы в конкретные требования
к нашей реализации.

---

## Причина бага №1: обрыв диктовки

**Это не таймаут.** В `%APPDATA%\Wispr Flow\logs\main.log`:

```
[warn] [Keyboard Service] Release event while dictation is in Initialized state,
       dismissing dictation.
```

Десять срабатываний, вместе с `[Dictation Tracing] finish(cancelled) called but no
active span` (тоже 10). Лог нативного хелпера подтверждает дисбаланс: **234 «Starting
dictation» против 244 «Stopping dictation»** — ровно десять остановок без старта.

**Механизм.** Отпускание кнопки приходит, когда машина состояний ещё в `Initialized`
и не дошла до `Recording`. Отпускание трактуется как отмена вместо остановки.

**Почему у заказчика это чаще.** Он использует голую Mouse 4 (код 4099) без
модификатора. У клика нет задержки удержания модификатора, которая маскирует окно
инициализации. Голая кнопка мыши — худший случай для этой гонки.

**Что усугубляет.** В логах `[IPC] Slow handler: KeypressEvent took 51–61ms` примерно
50 раз, `High system CPU load detected` 15 раз, `Audio lag exceeded 2s` со значениями
до **26,6 секунды**. Медленная обработка нажатия расширяет окно гонки.

**Подтверждение, что дело не в длительности:** настройки максимальной длительности для
обычной диктовки нет вообще (есть только `notetakerMaxRecordingLength: "2hr"` для
записи встреч), а в базе присутствует расшифровка длиной 368 секунд.

### Требование к BillyTalk

Событие отпускания **ставится в очередь**, а не обрабатывается, пока рекордер
фактически не достиг состояния Recording. Обработчик хука обязан только положить
событие в очередь и немедленно вернуться.

---

## Причина бага №2: вставка не в то поле

Вставка построена на буфере обмена с сохранением и восстановлением, поверх UI Automation.

Из лога хелпера:

```
Saved focused element at dictation start | hasFocusedElement=True     (234 раза, ни разу False)
Not refocusing textbox | currentStatus=..., hasSavedTextbox=...
Not refocusing textbox | currentStatus=Unknown, hasSavedTextbox=False
```

**Механизм.** Элемент фокуса надёжно сохраняется на старте. Но на вставке приложение
**сознательно не возвращает фокус** к сохранённому элементу, и вставка уходит туда, где
фокус оказался к моменту завершения.

При этом `[AppChangeMonitor] Detected app change to:` срабатывает постоянно —
`explorer` 582 раза, `putty` 353, `chrome` 257, `claude` 236. То есть приложение **знает**,
что фокус ушёл, и всё равно не возвращает его.

**Вторая половина проблемы — детект поля ввода хронически не работает:**

```
TextPattern is null, it is not supported                         322 раза
We don't know the app they're in has textboxes                   324 раза
Could not find a textbox, running fallback function              324 раза
Editable textbox pattern check | hasSomeTextPattern=False, editableCandidate=False
```

Перечисление состояний в бинарнике хелпера: `DefinitelyInTextbox` /
`DefinitelyNotInTextbox` / `Unknown`.

**Гигиена буфера обмена — отдельный источник боли.** Сотни строк `Found unknown clipboard
format, not in FormatsToIgnore or FormatsToRestore`: `Format_7`, `Format_17`,
`Ole Private Data`, `DataObject`, `CanIncludeInClipboardHistory`,
`Chromium internal source URL`, `FileGroupDescriptorW`. Они ведут явные списки форматов
для восстановления и постоянно натыкаются на необработанные.

### Требование к BillyTalk

Фокус захватывается в момент **нажатия**, проверяется перед вставкой, и при
расхождении фокус возвращается принудительно. Если вернуть не удалось — не вставлять
никуда, положить в историю и уведомить.

## Схема хоткеев — забираем целиком

**Главная находка. У Wispr кнопки мыши решены, и решены правильно.**

Клавиши и кнопки мыши живут в **одном числовом пространстве**. Мышь смещена на 0x1000:

```
mouse_middle : 4098      mouse4 : 4099      mouse5 : 4100
mouse6       : 4101      mouse7 : 4102      mouse8 : 4103
mouse9       : 4104      mouse10: 4105      mouse_aux : 65535
```

Сопоставление сырого индекса кнопки ОС в код: `{2: mouse_middle, 3: mouse4, 4: mouse5, ...}`
Отображаемые имена: `"Middle Click"`, `"Mouse 4"` … `"Mouse 9"`.

Сериализация хоткея — строка VK-кодов, склеенных через `+`, как ключ объекта:

```
prefs.user.shortcuts:
    "162+91"      => "ptt"                 (Ctrl+Win)
    "4099"        => "ptt"                 ← Mouse 4 заказчика, без модификатора
    "27"          => "dismiss"             (Esc)
    "160+164+90"  => "paste_last_text"
    "160+164+88"  => "copy_last_text"

prefs.cache.splitKeybinds:
    [{"shortcut":[162,91],"value":"ptt"}, {"shortcut":[4099],"value":"ptt"}, ...]

prefs.user.modifierShortcut: "164"          (одиночный модификатор для удержания)
```

Хранятся **обе формы** — и словарь, и массив.

Поток событий единый: `eventType: "mouse_event"` / `"mouse_event_release"`,
`inputType: "mouse"`. Клавиатура и мышь не разделены на два механизма.

**Это проверенный на проде дизайн. Копируем как есть, не изобретаем.**

## Выбор аудиоустройств — тоже забираем

Не один выбранный микрофон, а **ранжированный список с автопадением**:

```
rankedAudioDevices: [{deviceId, name}, ...]      порядок = приоритет
overrideAudioDeviceId: null                       null = автоматически
```

Плюс псевдоустройства `"default"` и `"communications"`, флаги `builtInAudio`,
`jabraWearDetectionEnabled`, `pairedBLEDeviceId` / `pairedBLEDeviceName`.

Отвалилась гарнитура — само упало на следующее устройство. **Прямо отвечает на
требование заказчика по выбору устройств.**

## Схема хранения истории — модель данных стоит перенять

**Файл:** `%APPDATA%\Wispr Flow\flow.sqlite`, SQLite, WAL, **662 МБ**.
ORM — Sequelize, 130 применённых миграций.

> ⚠️ **Изъято при подготовке к публикации.** Здесь были наблюдения о защите данных
> в чужом коммерческом продукте — см. примечание в конце файла.

**19 таблиц.** Основная — `History`, 2482 строки, **61 колонка**. Существенные:

```
asrText  formattedText  editedText  serverFinalizedText  pastedText  toneMatchedText
audio (BLOB)  screenshot (BLOB)  opusChunks (JSON)
app  url  micDevice  duration  speechDuration  numWords  numWordsCorrected
numDictionaryReplacements  e2eLatency  clientNetworkLatency
language  detectedLanguage  averageLogProb
axText  axHTML  textboxContents  userEditMetaData (JSON)
defaultAsrText  fallbackAsrText  usedFallbackAsr  fallbackLevel  *DivergenceScore
```

**Три приёма, которые стоит взять:**

1. **Хранить одновременно сырую расшифровку, отформатированный текст и правку
   пользователя**, и считать между ними расхождение. Это замкнутая петля обратной связи
   по качеству форматирования — видно, что именно модель портит.
2. **`axText` / `axHTML` / `textboxContents`** — контекст доступности целевого поля
   сохраняется вместе с каждой диктовкой. Пригодится и для отладки вставки, и для
   версии 3.0.
3. **Разная политика хранения для текста и звука** (ниже).

**Измеренная политика хранения:**

| Что | Факт |
|---|---|
| Строки расшифровок | 20.04.2026 → 17.07.2026, **ничего не удаляется** |
| Звук | только 616 строк, 04.07 → 17.07, **264 МБ** — то есть ~14 дней и удаляется |
| Скриншоты | 0 строк (разрешение не выдано) |
| Статусы | `formatted` 2335, `raw_transcript` 103, NULL 37, **`error` 6**, `empty` 1 |
| Длительность | мин 0,2 с · средняя 12,6 с · **макс 368 с** |
| Свободные страницы | 71601 (~293 МБ) — **VACUUM не делают никогда** |

**Урок:** текст вечно, звук ограниченно, и **обязательно уплотнять базу** — иначе
получим их 662 МБ с 293 МБ мусора внутри.

## Прочие настройки — бесплатный список требований

`%APPDATA%\Wispr Flow\config.json`, 34 КБ, открытый JSON, около 960 ключей.

**Обработка текста:** `defaultTranscriptionFormat` («Apply AI formatting»),
`appTranscriptionFormats[]` — **массив переопределений под конкретные приложения**,
`aiFormatting`, `styleDetectionEnabled`,
`personalizationStyles{work, email, personal, other, unset}`,
`autoCleanup.level`, `pressEnterCommandEnabled`, `useAxContext`,
`shouldOCRScreenCapture`, `typingLocations[]`.

**Словарь:** `shouldAutoLearnWords`, `dictionarySort`, отдельная таблица `Dictionary`
(15 строк, 16 колонок).

**Языки:** `selectedLanguages[]`, `usedLanguages[]`, `effectiveLanguages`,
`uiLocale` (у заказчика `ru`), плюс `detectedLanguage` на каждую расшифровку.

**Разрешения:** `prefs.permissions = [{name, clicked, granted}]` для `accessibility`,
`microphone`, `screen-capture` — модель «спросили / выдали» отслеживается явно.

**Приватность:** `localDataPolicy: "store_normally"`, `cloudSync: false`,
`shareUsageData: false`, `internalDataSharingEnabled: false`.

## Архитектура — на что смотреть, что не повторять

**Электрон, 11 процессов, ~1,14 ГБ памяти.** Десять `Wispr Flow.exe` плюс один
нативный `Wispr Flow Helper.exe` (44,9 МБ) — слой интеграции с ОС: UI Automation,
хуки клавиатуры и мыши, буфер обмена. В его строках `IUIAutomationElement`,
`INPUT/MOUSEINPUT/KEYBDINPUT`, `IMouseButtonReleaser`, `AppChangeMonitor`,
`RawInputMouseButtonCounts`.

Разделение «электронная морда + нативный хелпер» **подтверждает наше решение**
отделить ядро от интерфейса. Они пришли к тому же разбиению.

Но: в конфиге есть счётчики `HelperLoadingError`, `HelperNotReady`,
`HelperPersistentFailure` — с падениями хелпера они явно воюют. Плюс сторож
`gpu-guard.json` с автоперезапуском при всплесках GPU.

**Ни службы, ни задачи планировщика, ни драйвера, ни фильтра.** Только пользовательские
хуки и UI Automation. Наш подход тот же.

**Установка:** `%LOCALAPPDATA%\WisprFlow\`, per-user, без администратора, ~1,6 ГБ.
Squirrel.Windows, стадийная раскатка по проценту пользователей.

**Автозапуск — ярлык в папке автозагрузки**, не ключ реестра:
```
%AppData%\Microsoft\Windows\Start Menu\Programs\Startup\Wispr Flow.lnk
→ C:\Users\<...>\AppData\Local\WisprFlow\Wispr Flow.exe      (без аргументов)
```
Запускается **версионно-независимый стаб**, а не бинарник конкретной версии — поэтому
обновления Squirrel не ломают ярлык. Окно прячется внутри приложения, а не флагом
командной строки. Приём хороший, забираем; сам механизм у нас будет через `HKCU\...\Run`.

**Хранение токена.** Наш ключ и refresh-токен идут в **диспетчер учётных данных
Windows**, под защитой DPAPI. Без вариантов.

> ⚠️ **Изъято** — здесь были наблюдения о том, как хранит учётные данные чужой продукт.

**Потоковая расшифровка.** Wispr использует WebSocket-соединение со своим API, поэтому
текст появляется по мере речи. **У Groq потоковой расшифровки нет — только
запрос-ответ.** Это единственное место, где мы объективно проигрываем по ощущаемой
скорости, и это надо учитывать, сравнивая впечатление, а не миллисекунды.

> ⚠️ **Изъято** — перечень бэкенд-вендоров чужого продукта.

## Ответственное раскрытие

При разборе установленного продукта попались наблюдения о том, как он хранит данные
пользователя. Они **изъяты из этого файла перед публикацией репозитория**: публиковать
такое до уведомления вендора некорректно, независимо от того, что продукт коммерческий
и конкурирующий.

Если решим сообщить — писать на security-контакт Wispr AI, приложив конкретику,
и дать разумный срок до любой публикации. До тех пор наблюдения остаются в приватных
заметках.

**Границы разбора соблюдались с самого начала:** читались только схема, конфигурация
и собственные логи приложения на своей машине, на своих данных. Не читались содержимое
расшифровок, значения токенов и защищённые DPAPI данные. Трафик не перехватывался,
установка не изменялась.

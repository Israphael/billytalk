## S0 — стек жизнеспособен

Полная связка ставится на Python 3.14 без единой проблемы:

```
wxPython 4.2.5 (wxWidgets 3.2.9)   sounddevice 0.5.5   soundfile 0.14.0
pywin32 312   numpy 2.5.1   pyinstaller 6.21.0
```

Проверено не установкой, а работой:

- `WS_EX_NOACTIVATE` применяется к wx-окну после создания через `SetWindowLongPtrW` — **подтверждено**, флаг читается обратно. Неактивируемая плашка реализуема.
- **Кодирование FLAC внутри процесса: 9 мс** на 19,4 секунды аудио (против 152 мс через запуск ffmpeg, как предсказывала разведка). Внешний процесс не нужен.
- `win32clipboard` и `win32cred` импортируются.

⚠️ **Обязательно `SetWindowLongPtrW`, не `SetWindowLongW`,** и с явными `restype`/`argtypes`
типа `LONG_PTR`. Иначе то же 64-разрядное обрезание, что уже описано в `02-хоткеи`.

## S0b/S0c — ловушка насоса сообщений + отмена нативного помощника

### Ловушка: `PeekMessage` не получает колбэки хука

Цикл вида

```python
while ...:
    while u.PeekMessageW(byref(msg), None, 0, 0, PM_REMOVE):
        ...
    time.sleep(0.002)
```

даёт **ноль событий за 30 секунд активного движения мыши**. При этом
`SetWindowsHookExW` вернул валидный дескриптор, `GetLastError` — ноль.
Всё выглядит исправным.

Работает **только блокирующий `GetMessage`**, выход через
`PostThreadMessageW(tid, WM_QUIT, 0, 0)` с другого потока.

### Измерение стоимости: нативный помощник не нужен

90 секунд активного движения:

```
события          : 11 639   (129/с в среднем, ~460/с на пиках)
колбэк p50       : 1,7 мкс
колбэк p99       : 14,6 мкс
колбэк МАКСИМУМ  : 497 мкс      ← бюджет 1 000 000 мкс
процессор        : 0,03% одного ядра
```

Обзор предсказывал «до 1000 колбэков в секунду и постоянную нагрузку на GIL».
Реальность — 129 в секунду и три сотых процента ядра. **Запас до фатального порога
в 2000 раз.**

**Решение о выносе хуков в нативный процесс отменено.** Подробности —
`decisions/14-хук-остаётся-в-python.md`.

## S3 — перезагрузка PortAudio работает

```
sd._terminate(); sd._ffi.dlclose(sd._lib)
sd._lib = sd._ffi.dlopen(sd._libname); sd._initialize()
```

Отрабатывает за **42 мс**, не падает, поток после неё открывается нормально.
Штатный `_terminate()/_initialize()` без перезагрузки библиотеки список устройств
не обновляет — подтверждено.

⚠️ Перезагрузка **под живым потоком** тоже не уронила процесс. **Это не значит, что
она безопасна** — выжить один раз не равно быть корректным: код выгружается из-под
работающего потока обратного вызова. Оставляем правило: **останавливать поток перед
перезагрузкой**.

## S5 — задержка расшифровки

### aimlapi не подходит

Заказчик предложил использовать Groq через агрегатор aimlapi. Проверка:

| Проверка | Результат |
|---|---|
| Есть ли Groq | **нет ни под каким именем** |
| Архитектура | `POST /v1/stt/create` → опрос `GET /v1/stt/{id}` — **асинхронная очередь** |
| deepgram/nova-3 | **5638 мс** медиана |
| deepgram/whisper-large | **9615 мс** медиана |
| openai/gpt-4o-transcribe | HTTP 400, другие параметры |
| Цена лучшей модели | $0,26–0,46/час против $0,04 у Groq |

Отправка сама по себе занимала 2,6–3,5 с на 340 КБ — это их ingest, не канал:
измеренная отдача машины **~8 Мбит/с**.

**Провал по всем четырём осям сразу** — доступность модели, задержка, цена,
дополнительная третья сторона в цепочке аудио. Как запасной провайдер за интерфейсом
рассматривать можно, как основа режима 2 — нет.

⚠️ **Cloudflare режет `urllib` по сигнатуре клиента** (403, код 1010). С заголовком
`User-Agent` проходит. Учесть: наш HTTP-клиент обязан представляться.

### Groq напрямую — в бюджет

19,4 секунды речи, `whisper-large-v3-turbo`:

```
холодное соединение : 623 мс   (медиана 3 прогонов)
прогретое           : 368 мс   (медиана 5 прогонов)
выигрыш от пула     : 255 мс на каждую диктовку
```

Бюджет спецификации был 500–800 мс. **Вписались.**

Стоимость установки соединения измерена отдельно:
`api.groq.com` — dns 121 мс, tcp 367 мс, tls 540 мс нарастающим итогом.
**Держать прогретый клиент — обязательное требование, а не оптимизация.**

### FLAC против WAV: урок про выборку

Первый замер на **3 прогонах** дал «WAV быстрее FLAC на 183 мс» и едва не отменил
решение разведки использовать FLAC.

Повтор на **8 перемежающихся парах** (чередование гасит дрейф сети):

```
FLAC 339 КБ : 376 мс
WAV  604 КБ : 415 мс      → FLAC выигрывает 39 мс
```

**Первый результат был шумом.** FLAC остаётся. Записано как процедурное правило:
три измерения — не измерение, сравнения делать чередованием.

### Русский язык и параметр `prompt`

Синтезированная русская фраза с техническими терминами:

| Произнесено | Услышано без подсказки | С подсказкой |
|---|---|---|
| «БиллиТолк» | **BillyTalk** ✓ | BillyTalk ✓ |
| «впэ эн» | **VPN** ✓ | VPN ✓ |
| «эс эн ай» | **SNI** ✓ | SNI ✓ |
| «реалити» | реалити | **Reality** ✓ |
| «прод» | **прот** ✗ | **прот** ✗ |

Выводы:

1. **Whisper сам раскрывает произнесённые по буквам аббревиатуры** в латиницу.
   Словарь для них не нужен — это меняет ожидания от функции.
2. **`prompt` работает** для имён собственных: «реалити» → «Reality» только с ним.
3. **Обычные слова подсказкой не чинятся** — «прод» устойчиво слышится как «прот».
   Это и есть работа детерминированной замены.
4. ⚠️ **Подсказка портит пунктуацию.** С голым списком через запятую из ответа
   **пропали точки** — Whisper скопировал стиль подсказки как «предшествующего
   контекста». Разведка предупреждала теоретически, теперь подтверждено.

   → **Промпт формировать нормальным предложением с терминами внутри, а не списком.**

Английский образец, для сравнения: «BillyTalk» без подсказки слышится как
*«Bilitaug»*, с подсказкой — правильно. Латинские имена собственные страдают сильнее
русских.

## S6 — сторож живости хука

Тихое снятие хука системой изнутри не обнаруживается, нужен активный сторож.
Проверены две схемы, **обе работают**:

### A. Синтетическое эхо

`SendInput` с меткой в `dwExtraInfo`, хук ищет метку.

```
dx=0  : SendInput вернул 1 (успех), хук не увидел НИЧЕГО
dx=±1 : эхо за 0,8–1,1 мс
```

⚠️ **Нулевое смещение мыши система молча отбрасывает, при этом `SendInput` рапортует
успех.** Смещение обязано быть ненулевым; для незаметности — на один пиксель туда
и обратно.

### B. Расхождение с `GetLastInputInfo`

Сравнение системного времени последнего ввода с временем последнего события,
которое видел наш хук.

```
хук жив   : расхождение 0 мс
хук мёртв : расхождение 235 мс
```

Ничего не инжектирует, стоит один вызов API.

**Решение: связка.** Расхождение — постоянный дешёвый индикатор; эхо — подтверждение
перед объявлением хука мёртвым и переустановкой. Это исключает ложные срабатывания,
когда пользователь просто не трогал мышь.

## Сквозная находка: возвращаемое значение не есть доказательство

За один день это встретилось **три раза**:

1. `SetWindowsHookExW` вернул валидный дескриптор — событий нет (не тот насос сообщений).
2. `SendInput` вернул 1 при нулевом смещении — события не было.
3. Из разведки: `SendInput` рапортует успех вставки, которой не произошло
   (открытый 10 месяцев баг Whispering, 5–10% потерь).

**Правило для проекта: в API пользовательского ввода Windows успешный код возврата
не подтверждает, что действие состоялось. Проверять наблюдаемый эффект.**

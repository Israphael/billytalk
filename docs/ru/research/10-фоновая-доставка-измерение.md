# Спайк S7: доставка текста без вывода окна на передний план

**Дата:** 2026-07-19 · **Железо и окна заказчика**
**Скрипт:** `spikes/s7_background_delivery.py` · **Сырой лог:** `10a-background-delivery-raw.log`

## Зачем

Спайк S1 измерил принудительный возврат фокуса: **25% успеха**. Раздел доставки был
перестроен на буфер обмена как основной путь.

Затем разведка по проекту `trycua/cua` показала третий маршрут, которого не было
в нашем рассмотрении: **доставка сообщениями напрямую в целевое окно, без вывода его
на передний план.** У них это режим по умолчанию, названный `Background`, документирован
как «только PostMessage и UIA, никогда не выводит окно вперёд».

Если работает — текст попадает туда, где пользователь начал диктовать, независимо
от того, где сейчас фокус.

## Что проверялось

Три маршрута в один и тот же захваченный контрол, окно на передний план не выводится:

| Маршрут | Механизм |
|---|---|
| `WM_PASTE` | штатное сообщение вставки |
| `WM_CHAR` | посимвольная отправка в очередь контрола |
| `PostMessage Ctrl+V` | синтез сочетания клавиш в очередь цели |

Контрол в фокусе определялся через `GetGUIThreadInfo(tid).hwndFocus`.

## Результат

**Фоновая доставка работает частично.** Оценка заказчика по наблюдению: *«Word
не принимает, остальное кое-как срабатывает»*.

| Приложение | Класс контрола | Результат |
|---|---|---|
| **Word** | `_WwG` | **не принимает** |
| Chrome, Claude, Edge | `Chrome_WidgetWin_1` | срабатывает частично |
| Блокнот (Win11) | `RichEditD2DPT` | срабатывает частично |
| Telegram | `Qt51519QWindowIcon` | срабатывает частично |
| PowerPoint | `mdiClass` | до проверки не дошло |

**Подтверждено твёрдо: ни один из трёх маршрутов не украл фокус.** Ноль случаев
за все попытки. Это и было главным требованием к механизму.

Word отказывает закономерно: `_WwG` — собственная реализация ввода Microsoft,
посланные сообщения она не обрабатывает.

## Ошибка методики, которую стоит запомнить

**Автоматическая проверка не сработала ни в одном приложении.** Я читал контролы
через `WM_GETTEXT` — он не работает ни в новом Блокноте Windows 11 (`RichEditD2DPT`
это XAML/Direct2D, а не классический RichEdit), ни в Word, ни в Chromium.

При этом в спецификации уже было записано «UIA только на чтение», и именно UIA
следовало использовать для проверки. Спайк проверял механизм, которым сам же
не рекомендовал пользоваться.

Из 30 нажатий до проверки доехали 4: остальные отброшены, потому что заказчик
не успевал переключить окно за 4 секунды или возвращался обратно. **Окно ожидания
надо было делать больше, а условие мягче.**

Итог: механизм оценён **наблюдением заказчика, а не измерением**. Для решения этого
достаточно, но числа у нас нет, и выдавать оценку за измерение нельзя.

## Решение

**Фоновая доставка добавляется ступенькой в лестницу, но не становится основным путём.**

```
1. WriteHistory + WriteClipboard        ← безусловно, как и было
2. фокус не уходил   → вставить на месте
3. фокус ушёл        → а) одна попытка вернуть фокус        (25%)
                       б) фоновая доставка сообщениями      (частично, зависит от приложения)
                       в) громкий сигнал + Ctrl+Alt+Z       ← штатный исход
```

Ступень «б» стоит немного, иногда спасает вставку и **гарантированно не крадёт фокус**.
Но «кое-как» — не то основание, на котором строят основной путь.

**Реестр правил по приложениям получает третью колонку:** для каких классов окон
фоновая доставка вообще пробуется. `_WwG` (Word) — сразу пропускать, не тратить время.

Список пополняется по мере накопления наблюдений, по умолчанию — не пробовать.

## Что осталось непроверенным

- **Какой именно из трёх маршрутов срабатывал.** Заказчик наблюдал результат, а не
  механизм. Определяется дешёвой доработкой спайка с проверкой через UIA
- Ведёт ли `WM_CHAR` к посимвольной вставке, видимой пользователю как набор
- Поведение в PuTTY и mintty — до проверки не дошло, а это основные окна заказчика
- Синтетическая инъекция указателя (`CreateSyntheticPointerDevice`), которую использует
  `cua` — принципиально другой механизм, не проверялся вовсе

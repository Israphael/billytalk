# Разведка: голосовые командные центры

**Дата:** 2026-07-19
**Зачем:** амбиция «командного центра» — управление компьютером голосом, а не только
диктовка. Нужно понять, что в этой области уже решено, что мертво, и что из чужой
архитектуры можно взять, не нарушив лицензию.

**Метод:** обход репозиториев, исходников и трекеров. Читались файлы, а не описания
проектов: заявления в README расходятся с содержимым чаще, чем хотелось бы.

---

## Итог одной страницей

Область выглядит единой, а на деле это **три разные родословные**, которые постоянно
смешивают. Только одна из них про наш случай.

| Родословная | Состояние | Отношение к нам |
|---|---|---|
| **A. Классическое голосовое управление** (Talon, Dragonfly, Numen) | живо, но каждый проект держится на одном человеке | **вся полезная архитектура здесь** |
| **B. Ассистенты умных колонок** (Mycroft, Snips, Rhasspy) | кладбище; выжившие ушли в Linux и лампочки | не наша ветка, брать нечего |
| **C. LLM-агенты рабочего стола** (UFO, UI-TARS) | живо, активно, но по метрикам далеко от человека | не для основного пути |

Три вывода, каждый меняет план:

1. **Никто в родословной A не делает классификацию намерений.** Все используют
   фиксированные конечные грамматики. Это не отсталость, это правильный выбор.
2. **Для зрячих без рук задача решена** — Talon плюс Cursorless превосходны и
   недооценены. **Для незрячих не решена**, и упёрлась не в распознавание.
3. **Повторяющаяся причина провала во всём поле — не точность распознавания.
   Пользователь не может выяснить, что можно сказать.**

## 1. Родословная A: классическое голосовое управление

Та самая, из которой стоит брать.

| Проект | Лицензия | Состояние |
|---|---|---|
| **Talon** | ядро проприетарное, набор команд `talonhub/community` — MIT | **доминирует.** ~989 патронов на Patreon, коммиты в июле 2026, собственная модель Conformer, Windows/Mac/Linux, айтрекинг, распознавание шумов |
| **Dragonfly** | LGPL-3.0 | только сопровождение, bus factor 1, до сих пор `1.0.0-rc2` |
| **Caster** | LGPL-3.0 | ни одного релиза с 2019 |
| **Vocola2 / Natlink** | MIT | требует **снятую с продажи** платную лицензию Dragon |
| **kaldi-active-grammar** | **AGPL-3.0** | активно модернизируется |
| **Numen** | **AGPL-3.0** | только Linux |
| **Serenade** | Apache-2.0 | **мёртв с 23.07.2022.** Свежий `pushed_at` вводит в заблуждение: выглядит живым, не является |
| **VoiceAttack** | проприетарная | SAPI, игровой сегмент |

**Отдельный риск платформы.** Windows Speech Recognition выведена из эксплуатации
в сентябре 2024 и заменена на Voice Access. Всё, что построено на SAPI5, живёт
на заимствованном времени. Это касается не абстрактного «кого-то», а любого нашего
решения опереться на встроенные средства Windows.

**Общее свойство всех живых проектов родословной A: bus factor 1.** Держит их
не сообщество разработчиков, а пользовательская база, у которой нет альтернативы —
плюс, в случае Talon, killer app (Cursorless), превращающий вынужденность
в предпочтение.

## 2. Родословная B: ассистенты умных колонок — не наша ветка

Разбиралась не ради заимствований, а чтобы закрыть вопрос. Закрыт.

- **Mycroft** — убит патентным троллем (Voice Tech Corporation, патент США 10 491 679).
  Иск отозван, но компанию он обескровил; сворачивание в феврале 2023.
- **Snips** — куплен Sonos в ноябре 2019 и закрыт.
- **Rhasspy 2 и 3** — оба архивированы в октябре 2025.
- **Jasper, Kalliope, Almond** — мертвы.

Выжившие (Home Assistant / OHF-Voice, OVOS, Leon, Willow) — это Linux и ESP32, они
управляют лампочками и **явно отказываются от Windows**: `linux-voice-assistant`
прямо пишет «Windows is not supported». У OVOS **на 343 репозитория нет ни одного
навыка ввода с клавиатуры** — то есть в саму модель этих систем не заложено
взаимодействие с приложением на экране.

### Про «личного Джарвиса»

Репозитории с названием «Jarvis» на GitHub — в подавляющем большинстве учебные
и личные проекты: цепочки `if/elif` поверх готовых API, захардкоженные пути,
курсовые работы. Это нормальные учебные работы, просто не то, из чего можно что-то
взять. Два проекта реальные, и оба закрыты лицензией:

- `Priler/jarvis` — Rust, Vosk, **CC BY-NC-SA 4.0, некоммерческая**, только русский язык
- `isair/jarvis` — **только личное использование**

Итог: искать здесь нечего, и это стоило проверить, чтобы больше не возвращаться.

## 3. Родословная C: LLM-агенты рабочего стола

| Проект | Лицензия | Состояние |
|---|---|---|
| **Microsoft UFO** | MIT | активно сопровождается, дошёл до UFO³ |
| **UI-TARS-desktop** | Apache-2.0 | чистое зрение, без API доступности |
| **Anthropic computer-use-demo** | MIT | референс под Linux/Docker; в комплекте явные предупреждения о prompt injection и рекомендация подтверждения человеком для значимых действий |
| **Open Interpreter** | — | перезапущен на Rust, работу с рабочим столом делегирует наружу |

**Бенчмарки, ради которых стоило смотреть:**

| Бенчмарк | Человек | Агент |
|---|---|---|
| Windows Agent Arena, Navi (агент авторов бенчмарка, 2024) | **74,5%** | 19,5% |
| Windows Agent Arena, **Microsoft UFO² с o1** | 74,5% | **30,5%** |
| OSWorld | **72,36%** | текущий SOTA не подтверждён |

⚠️ Числа 19,5% и 30,5% часто путают. Первое — базовый агент из самой статьи Windows
Agent Arena, второе — лучший известный результат на том же наборе. Второе и есть
состояние области.

Репозиторий Windows Agent Arena при этом **не трогали с ноября 2024**. Двадцать месяцев
без коммитов — сквозь весь бум агентов. Любое заявление «мы SOTA на WAA» в 2026-м
меряется об замороженную линейку.

**Вывод не меняется от поправки: на LLM-управление компьютером основной путь
не строить.** Даже лучший результат — это две ошибки из трёх, а цена ошибки в нашем
случае — мусор, впечатанный в чужой документ.
Из родословной C берём **одну инженерную идею** (раздел 4), а не подход целиком.

## 4. Архитектура: что реально переиспользуемо

Это главная ценность разведки.

### 4.1 Грамматики вместо классификации намерений

**Никто в родословной A не классифицирует намерения.** Все до одного используют
фиксированные конечные грамматики (FST): почти 100% точности на закрытом словаре,
задержка меньше 100 мс.

Это правильный выбор, а не наследие. Когда осечка означает мусор, впечатанный
в редактор, «наиболее вероятное намерение» — неприемлемая семантика. Нужен ответ
«такой команды нет», а не «похоже, вы имели в виду».

### 4.2 Два распознавателя параллельно (Numen)

Numen держит **два экземпляра Vosk одновременно**:

- `cmdRec` — создан через `vosk.NewRecognizerGrm`, ограничен грамматикой
- `transRec` — создан с `nil`, свободная диктовка

Грамматика перезагружается на ходу через `cmdRec.SetGrm()`. Простая и рабочая схема:
команды и диктовка не конкурируют за один декодер.

### 4.3 Грамматика, собираемая в рантайме (`speech-to-phrase`) — лучшая найденная идея

`speech-to-phrase` (Apache-2.0) **строит FST для Kaldi во время работы** из живых имён
сущностей пользователя, а затем делает нечёткую коррекцию, подтягивая результат
обратно к точному имени.

Формулировка задачи в его README и есть суть:

> **какую из известных мне фраз сказал пользователь** — вместо «что было сказано»

Это переворот постановки. Не «распознай речь, потом пойми», а «выбери из закрытого
множества, которое я только что построил». Единственная лучшая идея, найденная
в этой разведке.

### 4.4 Диктовка внутри команды, а не отдельным режимом

Все выжившие проекты делают одинаково: свободный текст — это **типизированная дыра
внутри грамматики команды**, а не отдельный режим.

```
"search [for] <text>"   →   Dictation("text")
```

FST точно знает, где дыра открывается и где закрывается. **Режима, в котором можно
забыть, что ты в нём находишься, просто не существует.** Модальное переключение
проигрывает встроенному захвату — это подтверждается не рассуждением, а тем, что
переключение не выжило нигде.

### 4.5 Словарь Talon: неанглийский намеренно

Talon называет буквы `air / bat / cap / drum / each / fine / gust`, а команды —
`slap`, `pace`, `crunch`. Это **выдуманные фонетически различимые токены**, которые
не сталкиваются с обычной речью.

Плюс в грамматики заранее вшиты альтернативы, поглощающие типичные ошибки
распознавания:

```
(pace | paste) (that | it)
```

**Проблема неоднозначности здесь частично не решается классификацией, а растворяется
конструкцией словаря.** Дешевле придумать слово, которое не с чем спутать, чем
научиться разбирать путаницу.

### 4.6 Слой абстрактных действий (Talon)

Файлы `.talon` вызывают `edit.paste()`. Каждый контекст приложения реализует это
по-своему. Одна и та же произнесённая команда делает правильную вещь в VS Code,
в Vim и в браузере. Разделение «что сказал» и «как это исполняется здесь» — то,
чего нет ни в одной игрушке из родословной B.

### 4.7 Лучшие интеграции вообще обходят синтез ввода

`cursorless-dev/command-server` (MIT) — файловый RPC: пишет `request.json`, читает
`response.json`, управляет VS Code **по идентификатору команды со структурированными
аргументами**. Ни нажатий клавиш, ни координат.

Тот же принцип у Microsoft UFO — многоуровневое исполнение
в `ufo/automator/ui_control/{inspector,controller}.py`:

1. паттерны управления UIA
2. обёртки pywinauto
3. **синтетический ввод — только как задокументированное последнее средство**

Правило, которое стоит принять как своё: **синтез клавиатуры и мыши — это дно
иерархии, а не её основа.**

### 4.8 Недоиспользованный модуль Dragonfly

В Dragonfly есть `dragonfly/accessibility/` (`ia2.py`, IAccessible2), про который
почти никто не знает. Он даёт **семантическое нацеливание на текст**:

- `is_editable_focused()`
- `move_cursor(text_query, position)`
- `select_text(text_query)`

С мягкой деградацией, когда приложение не поддерживает IA2. Это прямо соседствует
с нашей задачей возврата фокуса и правки уже вставленного текста.

### 4.9 Шумы как первоклассный канал ввода

Talon и Numen **независимо друг от друга** реализовали распознавание неречевых звуков
(хлопок, шипение, выдох) как полноценные события ввода.

Почему это умно: канал с нулевой задержкой и нулевой неоднозначностью, не стоящий
ни одного цикла распознавания. Дискретное действие, которое невозможно спутать
со словом.

## 5. Узел доступности — самый важный раздел

Здесь область упирается в стену, и важно понимать, в какую именно.

### 5.1 Единственная серьёзная попытка

`sight-free-talon` (C-Loftus, **GPL-3.0**) — единственный серьёзный заход
на голосовое управление для незрячих.

- последний коммит — **апрель 2025**
- открытый issue **#47 с октября 2025: дополнение не работает с NVDA 2026 64-bit**
- **один мейнтейнер**

То есть флагманский проект области **сломан относительно актуальной NVDA, и починить
его некому.**

### 5.2 Философия, с которой трудно спорить

Документ проекта прямо отвергает рамку «приспособления». Автор хочет, чтобы
инструмент воспринимался как

> захватывающее улучшение, открывающее новые формы взаимодействия человека
> с компьютером

а не как нехотя выданная уступка. Это правильная рамка, и её стоит унаследовать
независимо от кода.

### 5.3 Что он реально даёт

- `echo_dictation` — **проговаривает обратно распознанное**, потому что увидеть
  появившийся текст нельзя
- `echo_context`
- `tts_via_screenreader`
- `braille_output`
- `sound_on_keypress` и `disable_keypresses` — описаны автором как **средства
  безопасности** против «ошибок распознавания, способных вызвать непреднамеренные
  действия»

Управление NVDA идёт через `nvdaControllerClient64.dll`:
`nvdaController_speakText`, `brailleMessage`, `cancelSpeech`.

### 5.4 Критическое ограничение

**IPC с NVDA используется, чтобы читать и переключать настройки NVDA, а не чтобы
получать содержимое экрана.**

Система **никогда не узнаёт, что на экране**. Сказать «нажми кнопку Отправить»
и чтобы это сработало по имени — по-прежнему нельзя. Обратная связь есть,
восприятия нет.

### 5.5 Две коронные жемчужины Talon доказывают структурный разрыв

- **Cursorless** размечает токены цветными «шляпами»
- **Rango** (MIT) накладывает буквенные метки на ссылки

Оба превосходны, оба MIT, и **оба полностью непригодны без зрения**. Визуальные
подсказки здесь не оформление, **они и есть механизм взаимодействия**: пользователь
называет то, что видит.

Это и есть форма разрыва. Лучшее в голосовом управлении построено на предпосылке,
которую незрячий пользователь не удовлетворяет.

### 5.6 Два issue, которые стоит выписать отдельно

**#41 — «Can't have TTS independent from Talon trying to interpret voice commands
from TTS audio».** Собственная речь системы попадает в микрофон и распознаётся как
команды. **У них не решено.**

Это третий независимый источник той же проблемы: у нас уже есть жалоба пользователя
из Чили (NVDA примешивается к диктовке в Word) и пост про NVDA, пробивающуюся
в исходящий поток Zoom — см. `09-версия-3-разведка.md`, раздел 2.3. Три разных
контекста, одна физика.

**#12 — онбординг инструмента доступности сам недоступен.** Комментировать нечего.

## 7. Предлагаемый синтез

Три идеи, каждая существует по отдельности, и, судя по всему, **их никто не соединял**:

1. **Собирать живые метки UIA** из окна переднего плана — подход `inspector.py`
   из Microsoft UFO.
2. **Компилировать эти метки в грамматику в рантайме** — приём `speech-to-phrase`,
   `SetGrm` у Vosk, живая перезагрузка как у Numen, плюс нечёткая коррекция,
   подтягивающая результат к точному имени элемента управления.
3. **Проговаривать доступные действия обратно** через NVDA Controller Client.

Это замыкает петлю, которую Talon замкнуть не может: **незрячему пользователю
не нужно видеть метки, потому что система их зачитывает и одновременно ограничивает
распознавание ровно этими метками.** Почти 100% точности на закрытом множестве,
никакой LLM в горячем пути.

Оговорка честная: **покрытие UIA — это вся игра, и оно не в нашей власти.** Отлично
в Win32/WPF/WinUI/Office, отсутствует в DirectX, canvas-приложениях, многих
поверхностях Electron и играх. Там, где UIA пусто, синтез рассыпается, и никакая
архитектура это не чинит.

## 8. Честная оценка области

- **Для зрячих без рук — решено, но неизвестно.** Talon + Cursorless превосходны
  и недооценены.
- **Ассистенты умных колонок и «личный Джарвис» — кладбище.** Ходить туда незачем.
- **Для незрячих — трудно и по-настоящему не решено.** За задачу взялся
  компетентный человек, на лучшей платформе, с настоящим вкусом к дизайну —
  и остановился ровно у той же стены.
- **У каждого открытого проекта родословной A bus factor 1.** Живут они
  на безальтернативности плюс, у Talon, на killer app.
- **На LLM computer-use основной путь не строить.**
- **Повторяющийся провал всей области — не точность распознавания, а то, что
  пользователь не может выяснить, что можно сказать.** Любая наша работа здесь
  должна начинаться с этого, а не с движка.

## Что осталось непроверенным

- **Текущий SOTA на OSWorld** — цифра человека (72,36%) подтверждена, актуальный
  результат агентов нет.
- **Зависит ли Windows Voice Access от UIA** — вывод хорошо обоснован, но это
  вывод, а не подтверждённый факт.
- **Внутреннее устройство движка Talon и отсутствие в нём API доступности** —

## 2. Захват ввода

### Механизм

`WH_MOUSE_LL` и `WH_KEYBOARD_LL` на выделенном потоке ядра, цикл — **блокирующий
`GetMessage`**.

> ⚠️ Измерено: **`PeekMessage` в цикле со сном не получает колбэки хука вообще** — ноль
> событий за 30 секунд при валидном дескрипторе и нулевой ошибке. Выход из цикла —
> `PostThreadMessageW(tid, WM_QUIT, 0, 0)`.

Стоимость измерена: p50 1,7 мкс, максимум 497 мкс при бюджете 1 000 000 мкс, 0,03% ядра.

**Колбэк только кладёт событие в очередь.** Единственное синхронное решение — подавлять
или нет.

**Снимок для колбэка.** Один неизменяемый объект в модульной переменной;
переприсваивание имени атомарно в CPython, блокировка не нужна.

```
HookSnapshot: bound_codes: frozenset[int] · enabled: bool · recording: bool
```

Множество «какие нажатия мы подавили» — **состояние процесса, а не объекта хука**,
иначе переустановка сторожем породит осиротевшее отпускание.

⚠️ **Ссылка на объект `WINFUNCTYPE` держится на уровне модуля** весь срок жизни хука.
`gc.freeze()` от сборки его не спасает — freeze переводит в постоянное поколение только
уже существующие объекты.

Обязательно: `gc.freeze()` после инициализации; PyInstaller **`--onedir`**, оба
исполняемых файла **`console=False`**.

### Номер кнопки

```c
WORD xbtn = HIWORD(ms->mouseData);   // 1 = Mouse 4, 2 = Mouse 5
```

### Пространство кодов

Единое, схема Wispr Flow, мышь смещена на 0x1000: `4098` средняя, `4099` Mouse 4,
`4100` Mouse 5 … `4105` Mouse 10.

### Подавление

Подавляются только привязанные коды. **Для аккордов подавляется только конечная
клавиша**, модификаторы — никогда.

**Решение о подавлении принимается на нажатии. Отпускание подавляется тогда и только
тогда, когда было подавлено парное нажатие** — независимо от того, изменилось ли
состояние между кромками. Это правило **выше** правила «Esc только во время записи»:
Esc переводит машину на кромке нажатия, и без приоритета парности его отпускание
осиротело бы.

**При установке хука** опросить `GetAsyncKeyState` по всем привязанным кодам. Уже
удержанный код помечается чужим: его отпускание не подавляется и события машины не
порождает.

**Подавление снимается**, когда: диктовка выключена из трея · процесс не работает ·
машина в состоянии, из которого запись начаться не может. **Кнопка никогда не должна
означать «ничего не происходит»** — если диктовать нельзя, «Назад» обязан вернуться.

`WM_XBUTTONDBLCLK` не обрабатывается: двойные клики синтезируются ниже хука.

**Автоповтор** гасится флагом состояния.

### Привязки по умолчанию

| Действие | Умолчание | Режим |
|---|---|---|
| Диктовка | **Mouse 4** (4099) | `ptt` |
| Диктовка, запасная | **Ctrl+Alt+D** | `ptt` |
| Диктовка тумблером | **Ctrl+Alt+T** | `toggle` |
| Отмена | **двойной Esc** во время записи | — |
| Вставить последнее | **Ctrl+Alt+Z** | — |
| Скопировать последнее | **Ctrl+Alt+X** | — |
| Главное окно | **Ctrl+Alt+B** | — |

⚠️ **Одиночный Esc проходит в приложение.** Отмена — двойной Esc в пределах 400 мс.
Иначе Esc становится непригоден для своего обычного назначения на всё время диктовки:
закрыть подсказку автодополнения, свернуть модальное окно.

### Сторож живости

1. **Расхождение** `GetLastInputInfo.dwTime` с временем последнего события хука > 200 мс
   при активном вводе → подозрение. Сравнение **знаковое**: `dwTime` может уменьшиться,
   отрицательная разница означает «ввода не было», а не расхождение.
2. **Подтверждение только эхом**: `SendInput` с меткой в `dwExtraInfo`, смещение ±1
   пиксель туда и обратно. Жив ⟺ хук увидел метку за 50 мс.

⚠️ **Расхождение после эха не перечитывается.** `SendInput` обновляет `GetLastInputInfo`
и подставляет свой tick count — эхо само обнуляет расхождение и замаскировало бы мёртвый
хук. После эха время последнего события хука сбрасывается.

⚠️ **Во время запроса UAC эхо не слать** — оно не доходит до защищённого рабочего стола
и ложно объявит живой хук мёртвым. Хук и `GetLastInputInfo` в этот момент замолкают
одновременно, расхождения не возникает.

Переустановка выполняется **на потоке-владельце** через сообщение, множество подавленных
нажатий наследуется.

⚠️ Нулевое смещение мыши система молча отбрасывает при успешном коде возврата `SendInput`.

## 3. Долговечность

**Речь становится долговечной в момент остановки записи, до сетевого вызова.**

```
StopCapture
  ├─ 1. обрезать тишину, закодировать FLAC
  ├─ 2. записать файл в %LOCALAPPDATA%\BillyTalk\audio\{uuid}.flac
  ├─ 3. INSERT в историю со статусом pending_transcribe
  └─ 4. только теперь — сетевой запрос
```

Без этого падение процесса, спящий режим, перезагрузка Windows Update или обрыв сети
уничтожают диктовку целиком, вместе со всеми тремя путями отхода — они все растут
из записи в истории.

**При старте ядро сканирует** записи в нетерминальных статусах (`pending_transcribe`,
`pending_retry`) и ставит их в очередь повтора. Пользователь видит «N записей ждут
расшифровки».

### Сроки хранения аудио

| Состояние записи | Срок |
|---|---|
| Доставлена (`inserted`, `left_on_clipboard`, `focus_lost`, `verify_impossible`) | **1 час** от доставки |
| Ждёт расшифровки или повтора | **не удаляется** до доставки или явного отказа пользователя |
| **Сети нет дольше часа** | **все нерасшифрованные удерживаются, уборка приостановлена целиком** |

**Правило при отсутствии сети (решение заказчика).** Пока связь не восстановлена,
уборка аудио **не запускается вообще** — даже по доставленным записям. Причина: при
долгом офлайне очередь непрочитанных растёт, и любая уборка в этот момент увеличивает
риск потерять единственную копию.

Возобновление уборки — через 10 минут после первой успешной расшифровки.

**Индикация обязательна.** Пока сеть недоступна: значок трея в состоянии «офлайн»,
подсказка «N записей ждут связи», и **звуковой сигнал `stop` заменяется на `clipboard`**
— пользователь должен слышать, что текст не появится сейчас, а не обнаружить это
глазами через минуту.

Верхняя граница удержания — **500 записей или 2 ГБ**, дальше самые старые вытесняются
с уведомлением. Без потолка офлайн на неделю забил бы диск.

**Сборка мусора при старте:** файлы в `audio\` без ссылающейся строки удаляются.

**Питание.** `WM_POWERBROADCAST`:
- `PBT_APMSUSPEND` — принудительно финализировать запись, сохранить, закрыть аудиопоток,
  сбросить состояние подавления
- `PBT_APMRESUMEAUTOMATIC` — перечислить устройства, переустановить хуки безусловно,
  сбросить базовые отсчёты сторожа

**Завершение сеанса.** `WM_QUERYENDSESSION` — немедленно сохранить всё в полёте и
вернуть TRUE. `ShutdownBlockReasonCreate` только на время сохранения, никогда на время
сетевого вызова: Windows даёт около пяти секунд.

**Все таймеры и предохранители — на монотонных часах.** Скачок больше двойного
предохранителя трактуется как сон, а не как таймаут.

## 4. Машина состояний

Чистая функция `(state, event, now) → (new_state, [effects])`.

### Состояние — запись, не скаляр

```python
@dataclass(frozen=True)
class State:
    phase: Phase          # Idle|Initialized|Recording|Finalizing|Delivering|Failed
    enabled: bool
    pending_release: bool
    queue: tuple[QueuedPress, ...]        # максимум 3
    active: ActiveDictation | None        # seq, started_at, target_hwnd
    next_seq: int
```

### Эффекты

`StartCapture` · `StopCapture` · `CancelCapture` · **`PersistAudio`** ·
**`WriteHistory(seq, status)`** · `Transcribe` · `WriteClipboard` · `Insert` ·
`PlayCue(kind)` · `Notify(code)` · `EnqueueRetry` · `ReinstallHook` ·
`SetSuppression(bool)` · `Shutdown`

### Таблица состояние × событие

| Событие | Idle | Initialized | Recording | Finalizing | Delivering | Failed |
|---|---|---|---|---|---|---|
| Нажатие `ptt` | → Initialized, StartCapture | игнор | игнор | в очередь* | в очередь* | → Initialized |
| Отпускание `ptt` | игнор | **отложить** | → Finalizing, StopCapture, PersistAudio, WriteHistory | игнор | игнор | игнор |
| Нажатие `toggle` | → Initialized, StartCapture | игнор | → Finalizing, StopCapture, PersistAudio, WriteHistory | в очередь* | в очередь* | → Initialized |
| Отпускание `toggle` | игнор | **игнор** | игнор | игнор | игнор | игнор |
| Запасная привязка | → Initialized, StartCapture | игнор | игнор | в очередь* | в очередь* | → Initialized |
| **`CaptureStarted`** | — | → Recording, применить отложенное, PlayCue(start) | — | — | — | — |
| **`TranscribeOk`** | — | — | — | → Delivering, WriteClipboard, Insert | — | — |
| **`InsertOk`** | — | — | — | — | → Idle или следующий из очереди | — |
| **`InsertFailed`** | — | — | — | — | → Idle, PlayCue(clipboard), Notify | — |
| **`HistoryWriteFailed`** | — | — | продолжить, PlayCue(warn) | — | — | — |
| Двойной Esc | игнор | → Idle, CancelCapture, погасить отложенное | → Idle, CancelCapture, `cancelled` | → Idle, отменить запрос, `cancelled` | игнор | → Idle |
| Предохранитель | — | — | → Finalizing, **в историю, НЕ вставлять** | — | — | — |
| Ошибка микрофона | → Failed, Notify | → Failed, Notify | → Finalizing (что записано) | — | — | — |
| Ошибка сети | — | — | — | → Idle, EnqueueRetry, `pending_retry` | — | — |
| Хук умер | ReinstallHook | → Finalizing | → Finalizing | ReinstallHook | ReinstallHook | ReinstallHook |
| Сон | — | → Finalizing | → Finalizing | — | — | — |
| Выключение из трея | SetSuppression(off) | → Idle, CancelCapture | → Finalizing, **в историю, НЕ вставлять** | доработать | доработать | → Idle |
| Выход | Shutdown | CancelCapture, Shutdown | сохранить, Shutdown | **сохранить**, Shutdown | **сохранить**, Shutdown | Shutdown |
| Очередь полна | — | — | — | **отказ на нажатии**, PlayCue(reject) | то же | — |

\* очередь ставится **на нажатии**, максимум 3. Четвёртое **отклоняется немедленно
сигналом отказа, до того как пользователь начал говорить** — отказывать после речи
бесполезно.

**`Failed` транзиентно:** после `Notify` машина сама возвращается в `Idle` и снимает
подавление. Кнопка не должна залипать в «ничего не делает».

### Отложенное отпускание

Отпускание в `Initialized` **откладывается и применяется при `CaptureStarted`.**
Не отменяет диктовку. Правило действует **только для `ptt`**; в `toggle` отпускание
игнорируется во всех состояниях.

Это причина обрывов у Wispr Flow: `Release event while dictation is in Initialized
state, dismissing dictation`, дисбаланс 234 старта против 244 остановок.

### Порядок доставки

Одна активная запись. Расшифровки могут идти параллельно, **доставка строго по порядку
нажатия**. Отказавшая расшифровка **освобождает свою позицию** и уходит в отдельную
дорожку повторов, которая никогда не вставляет автоматически.

### Слишком короткое и пустое

Нажатия короче **250 мс от первого захваченного кадра** (не от нажатия) и клипы, где
ни один кадр не превысил порог, **не расшифровываются — но запись в историю создаётся**
со статусом `too_short` или `empty`. Молчит только сигнал и уведомление.

Без записи критерий приёмки нефальсифицируем: невозможно отличить «BillyTalk потерял»
от «я не нажимал».

### Инварианты

```
count(StartCapture) == count(StopCapture)      ← у Wispr 234 против 244
никогда двух одновременных захватов
никогда Insert без предшествующего TranscribeOk
WriteHistory всегда предшествует Insert
никогда WriteClipboard без предшествующего PersistAudio
```

⚠️ Инварианты проверяют модель. **В драйвере — рантайм-проверка:** открытый захват без
состояния `Recording` принудительно закрывается и доставляется. Настоящий разрыв 234/244
у конкурента вызван падениями и смертью хука, чего чистая функция не моделирует.

## 5. Звук

`sounddevice`, **16 кГц моно, `int16`**.

### Микрофон

Ранжированный список с автопадением. ⚠️ Список PortAudio заморожен на инициализации;
обновление — перезагрузка библиотеки (измерено 42 мс) **при остановленном потоке**.

Смена устройства во время записи: текущая запись **финализируется и доставляется**
на записанном. Склейка не делается.

Изменения ловятся через `WM_DEVICECHANGE` на скрытом окне верхнего уровня.

⚠️ Bluetooth: захват переключает профиль на HFP (моно 8–16 кГц), качество падает.
Предупреждать. Тракт просыпается 100–300 мс, **первые слоги теряются физически** —
держать входной поток открытым в паузе между диктовками, чтобы устройство не засыпало.

### Обрезка тишины

Кадры 20 мс. Порог = **4 × медианная среднеквадратичная амплитуда самого тихого дециля**.

⚠️ **Обязателен абсолютный нижний порог.** Адаптивный порог на клипе, целиком состоящем
из речи, ставит планку на 12 дБ выше собственного пола клипа — короткое «да», «нет»,
«ок» вырезается целиком. **Клип не отбрасывается никогда, если его пик превышает
−35 дБFS.**

Добивка 180 мс с каждой стороны. Хвост 200 мс после отпускания входит в добивку.

### Кодирование

**FLAC внутри процесса** через `soundfile` — 9 мс. Измерено: быстрее WAV на 39 мс
по полному циклу (8 перемежающихся пар).

### Сигналы

Пять различимых: `start` · `stop` · `clipboard` · `reject` · `error`.

**`clipboard` — громкий и неподавляемый**, это единственный надёжный канал сообщения
об исходе «текст остался в буфере»: уведомления глушит фокусировка внимания, плашку
пользователь отключает, значок трея Windows 11 прячет в переполнение.

Сигнал `start` играет **только после фактического открытия потока** — иначе это
подтверждение несостоявшейся записи.

## 6. Расшифровка

### Интерфейс

```
TranscriptionProvider:
  capabilities → {languages[], streaming, max_clip_seconds, supports_prompt, cost_tier}
  transcribe(AudioClip, TranscriptionOptions) → TranscriptionResult
  health() → Status
```

`TranscriptionResult` несёт **`billed_seconds`** с первого дня.

### Groq

```
POST https://api.groq.com/openai/v1/audio/transcriptions
model=whisper-large-v3-turbo   response_format=json
```

Измерено: **368 мс прогретым, 623 мс холодным**; установка соединения 540 мс.
**Пул прогретых соединений обязателен.** ⚠️ Клиент обязан слать `User-Agent` —
Cloudflare режет запросы без него.

Таймаут 30 с. Предел клипа 20 минут, при превышении отказ на нажатии.

### Ошибки

| Код | Поведение | Пользователю |
|---|---|---|
| Таймаут | в очередь повтора | «нет связи, повторю» |
| 401 | **не повторять** | «ключ недействителен» + в настройки |
| 429 | повтор по `Retry-After` | «лимит, жду N с» |
| 5xx, сеть | повтор 1→2→4→8 с, до 5 попыток | «сервис недоступен» |
| Пустой текст | запись `empty`, без сигнала | — |

Повтор автоматический, **результат никогда не вставляется сам** — уходит в историю.

### Язык

Задаётся явно, автоопределение выключено. **Шаг мастера** и **пункт меню трея**.

Измерено: Whisper сам раскрывает произнесённые по буквам аббревиатуры — «впэ эн» → `VPN`,
«эс эн ай» → `SNI`, «БиллиТолк» → `BillyTalk`.

### Промпт

⚠️ Измерено: **голый список через запятую портит пунктуацию** — из ответа пропали точки.
Промпт формируется **нормальным предложением** с терминами внутри, шаблон на язык,
термины из правых частей словаря, ценные — в конец, лимит 224 токена.

### Причёсывание

Отдельный ключ, отдельный переключатель, **по умолчанию выключено**. Отказ мягкий.
Хранятся оба варианта.

## 7. Словарь

Детерминированная замена после расшифровки, до причёсывания.

| Тип | Пример |
|---|---|
| `normalize` | `впс\|ВПС\|vps → VPS` |
| `replace` | `прот → прод` |

Сопоставление регистронезависимое, по границам слов **с `re.UNICODE`**, длинные правила
раньше коротких, не срабатывает внутри слова. Хранится в таблице БД, выгружается JSON.

> Стартовая таблица собирается из данных заказчика. Список из редакции 1 был вымыслом:
> `Remnawave`, `Reality`, `XTLS`, `SNI`, `Happ`, `Groq`, `Tauri` не встречаются
> в 2445 расшифровках ни разу. Реальные: `VPS` в четырёх написаниях, `VPN` в трёх,
> устойчивое «прот» вместо «прод».

## 8. Доставка текста

**Иерархия перевёрнута против редакции 2 по результатам измерения.**

### Порядок

```
1. WriteHistory        ← уже сделано в момент StopCapture (раздел 3)
2. WriteClipboard      ← БЕЗУСЛОВНО, до любых попыток вставки
3. фокус не уходил?  → вставить на месте
   фокус ушёл?       → одна попытка вернуть; не вышло → PlayCue(clipboard) + Notify
4. проверить результат, если возможно
5. восстановить буфер
```

**Буфер обмена — основной путь, а не откат.** Текст в нём есть всегда, независимо
от исхода вставки.

### Возврат фокуса

HWND захватывается **в момент нажатия**. Перед вставкой сверяется.

Ушёл — **одна попытка** голым `SetForegroundWindow`. Не вышло — исход `focus_lost`,
громкий сигнал, уведомление с готовым `Ctrl+Alt+Z`.

> **Измерено: 25% успеха** (`research/08`). `SetForegroundWindow` возвращает FALSE сразу
> почти везде — клик пользователя передаёт «кредит последнего ввода» другому приложению.
>
> ⚠️ **`AttachThreadInput` вычеркнут из проекта.** Не помог **ни в одном** из восьми
> испытаний, а сцепляет очереди ввода: зависание чужого потока вешает нас, при этом
> у хука бюджет в одну секунду.
>
> Severity ниже числа: при задержке Groq 368 мс надо успеть кликнуть в другое окно
> за треть секунды. Случай редкий — но обязан быть слышимым.

⚠️ **UIA только для чтения.** `TextPattern` писать не умеет вовсе, `ValuePattern.SetValue`
**заменяет значение поля целиком** и уничтожил бы уже набранный пользователем текст.
Единственное исключение: `ValuePattern` есть, `IsReadOnly` ложно и текущее значение пусто.

### Способ вставки

Буфер + `Ctrl+V`. Резерв — `SendInput` с `KEYEVENTF_UNICODE`.

**Клавиша через VK-код `0x56` без `KEYEVENTF_SCANCODE`** — переживает кириллицу,
AZERTY и DVORAK.

**Защита от зажатого модификатора:** опросить `GetAsyncKeyState`, ждать до 500 мс
отпускания. Не отпустились — не вставлять, исход `left_on_clipboard`.

### Буфер обмена

Одна сессия: `OpenClipboard` → `EmptyClipboard` → `CF_UNICODETEXT` → форматы исключения
→ `CloseClipboard`.

⚠️ **Счётчик последовательности — два разных снимка.** `seq_before` до `OpenClipboard`,
`seq_after` после `CloseClipboard`. Наша запись счётчик увеличивает — это подтверждение.
**Опорным для охраны восстановления служит `seq_after`, не `seq_before`.** Сверка
с `seq_before` отменяла бы восстановление всегда и уничтожала чужой буфер при каждой
диктовке.

`GetClipboardSequenceNumber` возвращает 0 при отказе доступа — **ноль не значение,
а отказ**: восстановление не выполняется.

⚠️ **Перечитать счётчик непосредственно перед `SendInput`.** Между нашей записью
и нажатием менеджер буфера или `Ctrl+C` пользователя могут подменить содержимое — иначе
мы вставим чужой текст.

Форматы исключения через `RegisterClipboardFormatW`, данные — **реальный
`GlobalAlloc(GMEM_MOVEABLE, 4)`, не `NULL`**.

⚠️ **Флаги исключения ставятся по цели, а не по исходу.** Ведётся список процессов,
где проверка вставки достоверна; только там текст исключается из журнала. Везде иначе
флаги не ставятся — иначе `Win+V` слепнет ровно там, где он единственная страховка,
потому что `verify_impossible` для основных окон заказчика это типовой исход.

Снимок — `CF_UNICODETEXT` плюс явный список по именам. **Перебор всех форматов
запрещён:** отложенная отрисовка заставляет владельца рисовать синхронно и вешает нас
на зависшем приложении.

`OpenClipboard` — до 10 попыток с откатом 10→200 мс. **Операция с буфером — глобальный
мьютекс** (доставка и вставка из истории делят последовательность).

### Проверка результата

| Условие | Исход |
|---|---|
| элемент недоступен, `TextPattern` пуст, чтение упало | `verify_impossible`, **молча** |
| текст найден после позиции каретки | `inserted` |
| значение не изменилось | `paste_failed`, сигнал и уведомление |

**Политика при `verify_impossible` (решение заказчика): молчать, но текст всегда
в буфере и в истории.** Пути отхода: `Ctrl+V`, `Win+V`, `Ctrl+Alt+Z`.

### Правила приложений

| Класс | Сочетание |
|---|---|
| Windows Terminal, Alacritty, новые сборки PuTTY | `Ctrl+Shift+V` |
| legacy `conhost` (cmd) | `Ctrl+V` |
| прочие | `Ctrl+V` |

⚠️ **Для окон терминального класса переводы строк заменяются пробелами перед записью
в буфер, `press_enter_after` принудительно выключен, а `Ctrl+Alt+Z` требует
подтверждения.** Вставка с `\n` в живую SSH-сессию **исполняет команду**, отмены там нет,
а заказчик держит по SSH инфраструктуру VPN.

### Защищённые поля и повышенные окна

`IsPassword`, `ES_PASSWORD`, `Credential Dialog Xaml Host` → не вставляем, исход
`blocked_secure`, сигнал и уведомление. **Молчаливое бездействие недопустимо.**

Окно с правами администратора: хук его не видит, вставка не пройдёт. Обнаружив —
**не тратить запрос к Groq**, сразу `left_on_clipboard`.

## 9. Вставить и скопировать последнее

`Ctrl+Alt+Z` — вставить, `Ctrl+Alt+X` — скопировать.

**«Последнее» = самое свежее по времени нажатия среди показанных пользователю.**
Прибытия из дорожки повторов исключаются: иначе семиминутной давности текст прилетит
по `Ctrl+Alt+Z` в PuTTY.

Старше 2 минут — требовать подтверждения. В уведомлении и подсказке трея показывать
первые ~40 знаков и возраст.

**Уведомление об отказе всегда несёт готовое действие**, а не констатацию.

## 11. Интерфейс

### Трей

Собственная реализация на ctypes (`wx.adv.TaskBarIcon` даёт только версию 3 и сам
обрабатывает `TaskbarCreated` — слой поверх дал бы двойной `NIM_ADD`).

Запуск **только в трей**, окно не создаётся. `NOTIFYICON_VERSION_4` с `NIM_SETVERSION`.
Идентификация по **`hWnd` + `uID`**, не `NIF_GUID`. Подсказка требует **`NIF_SHOWTIP`**.

Состояния значка: покой · запись · расшифровка · очередь · **остановлен** · ошибка.

`TaskbarCreated` — полный `NIM_ADD` + `NIM_SETVERSION`, обработчик идемпотентен.
⚠️ Окно **настоящее верхнего уровня** (`hWndParent = NULL`, не `HWND_MESSAGE`),
`RegisterWindowMessageW` **до** создания окна.

### Плашка

`WS_EX_NOACTIVATE | WS_EX_TRANSPARENT | WS_EX_LAYERED | WS_EX_TOOLWINDOW | WS_EX_TOPMOST`,
показ через `ShowWindow(SW_SHOWNOACTIVATE)`, **никогда `frame.Show()`**.
`SetWindowLongPtrW` с явными `LONG_PTR`.

**Живёт через `Finalizing` и `Delivering`** с видом «расшифровка» — иначе между
отпусканием и текстом пользователь ~400 мс не знает ничего и жмёт кнопку повторно.

⚠️ **Плашку нельзя отключить, пока пользователь ни разу не воспользовался
`Ctrl+Alt+Z`** — иначе он остаётся без единого канала обратной связи.

### Уведомления

⚠️ Без зарегистрированного AppUserModelID и ярлыка в меню «Пуск» тосты не показываются
вовсе, молча. ⚠️ Фокусировка внимания их подавляет.

**Поэтому уведомление никогда не единственный канал:** любой отказ дублируется звуковым
сигналом и состоянием значка.

## 12. Установщик, автозапуск, первый запуск

**NSIS `currentUser`**, `%LOCALAPPDATA%\Programs\BillyTalk\`, **без версии в пути**.

Автозапуск — `HKCU\...\Run`, запись при установке, ключ пишется напрямую.
⚠️ Проверка отключения — **`byte0 & 1`**, не `== 3`: встречаются `06`/`07`.
Удаление сносит и запись `StartupApproved`.

**Права обычные.** Повышенный процесс не сможет вставлять в обычные приложения.

### Мастер первого запуска

1. **Микрофон** — попробовать захват, при отказе кнопка в `ms-settings:privacy-microphone`
2. **Язык диктовки**
3. **Хоткей** — живой захват, предупреждение про «Назад»
4. **Отвязка «Назад» на уровне драйвера** — см. ниже
5. **Способ расшифровки** — см. ниже. Здесь же **предупреждение про интернет**
6. **Ключ Groq** — ссылка на `console.groq.com`, «бесплатно, без карты», диспетчер
   учётных данных, проверка запросом
7. **Проверка** — зажать, сказать, увидеть текст

### Шаг 5: способ расшифровки и предупреждение про сеть

**Предупреждение обязательно и формулируется прямо, без смягчения:**

> **BillyTalk расшифровывает речь в облаке.** Без интернета или на слабой связи
> диктовка **не будет работать**: записи сохранятся и расшифруются, когда связь
> появится, но текст в этот момент вы не получите.

Заказчик живёт в Бразилии — обрывы связи нередки, и человек, узнавший об этом
ограничении в самолёте, а не при установке, воспримет это как поломку.

**Проверка железа и предложение локального режима (1.0).** На этом же шаге приложение
оценивает машину — процессор, память, наличие подходящей видеокарты — и, **если железо
позволяет, предлагает поставить локальную модель сразу**:

> Ваш компьютер потянет локальную расшифровку. Она работает **без интернета
> и без ограничений**, а ваш голос не покидает машину. Скачивание — около N МБ.

Причина ставить это в мастер, а не ждать порога в 2000 слов: **пользователь, у которого
железо есть, вообще не должен нагружать наш API**, а офлайн-независимость решает
и проблему связи выше.

Если железо слабое — вариант **не показывается вовсе**, чтобы не отправлять человека
в режим, где будет хуже.

⚠️ В MVP-0 локального режима нет, поэтому шаг ограничивается предупреждением про сеть.
Проверка железа и предложение появляются в 1.0 вместе с самим режимом.

Затем уведомление и **предложение закрепить значок трея**.

Мастер пропущен без ключа → первая диктовка показывает «нужен ключ» с кнопкой,
а не молчит.

### Шаг 4: отвязка «Назад» в драйвере

Провести пользователя через снятие функции «Назад» с боковой кнопки в фирменной
программе мыши (Razer Synapse, Logitech G HUB и подобных), со скриншотами.

**Зачем:** наше подавление — единственная защита от того, что браузер уйдёт «Назад»
и потеряет заполненную форму. Но подавления нет при падении процесса, над окнами
с правами администратора и в зазоре при переустановке хука. **Отвязка в драйвере делает
любой такой сбой безвредным вместо разрушительного.**

Шаг необязательный, с честным объяснением цены и выгоды.

## 13. Ключи и приватность

Ключи — **диспетчер учётных данных Windows**, `BillyTalk/groq-api-key`,
`BillyTalk/polish-api-key`.

### Инвариант редактирования

1. `__repr__` и `__str__` чувствительных типов возвращают **константу
   `"<redacted:transcript>"`**
   ⚠️ **Не возбуждать исключение.** Исключение в `__repr__` ломает вывод assert'ов,
   отладчик и — критично — при срабатывании внутри обработчика ошибки превращает
   восстановимый отказ вставки в необработанное падение, которое по разделу 3
   уничтожило бы текст. Механизм приватности не имеет права терять слова пользователя.
2. `logging.Filter`, отбрасывающий записи с этими типами
3. Правило линтера в CI

**Дополнительно, помимо `logging`:**

- ⚠️ **Отчёты об ошибках Windows.** `WerAddExcludedApplication` для обоих исполняемых
  файлов при установке, `SetErrorMode`, обработчик верхнего уровня. Иначе полный дамп
  памяти с буфером аудио, расшифровкой и ключом уходит в `%LOCALAPPDATA%\CrashDumps`
  и, возможно, в Microsoft.
- ⚠️ **Сторонние логгеры.** `urllib3`, `http.client`, `requests`, `comtypes` явно
  прижимаются к `WARNING`; при старте проверяется `HTTPConnection.debuglevel == 0`.
- ⚠️ **Остатки SQLite** — `secure_delete=ON`, плюс действие «очистить историю и аудио».

**Никогда:** аудио, текст расшифровки и любая его подстрока, хеш или длина, заголовок
и содержимое целевого окна, содержимое буфера, нажатия сверх хоткея.
**База — не лог:** `target_app` (имя процесса) в базе допустим, в логе нет.

## 14. Архитектура

Два процесса: ядро (хуки, звук, машина, провайдеры, вставка, хранилище, скрытое окно,
трей) и интерфейс на wxPython. **В `HKCU\Run` прописано ядро.**

⚠️ **Канал:** `\\.\pipe\billytalk-{sid}-{session_id}` — только SID недостаточно,
пространство имён общемашинное, и один пользователь в двух сессиях получил бы коллизию.
**`FILE_FLAG_FIRST_PIPE_INSTANCE`**, **явный DACL только на текущего пользователя**
(по умолчанию Windows даёт чтение группе «Все», а по каналу идут расшифровки),
`PIPE_REJECT_REMOTE_CLIENTS`. Интерфейс сверяет `GetNamedPipeServerProcessId`
с путём образа.

**По каналу передаётся `{id}`, не текст** — интерфейс запрашивает содержимое отдельно.

**`history_export` принимает только путь, полученный из системного диалога выбора
файла**, и ядро его валидирует: иначе это примитив произвольной записи файла.

**Режим захвата хоткея** имеет таймаут 30 с и снимается при разрыве канала — иначе
падение интерфейса оставит кнопку подавленной без способа это исправить.

Схема конфигурации — как в редакции 2, плюс `usage.words_this_week` под режим 2.
Детали протокола, DDL и раскладка — в `03-harness-слой-для-сборки.md`.

## 16. Приёмка

**Долговечность**
- [ ] Убить ядро во время расшифровки → после перезапуска запись найдена и доставлена
- [ ] Отключить сеть → текст в истории, аудио **не удалено**, повтор при возврате связи
- [ ] Сон во время записи → после пробуждения запись сохранена, хук работает
- [ ] Аудио доставленного старше часа удалено; аудио нерасшифрованного живо

**Ввод**
- [ ] Удержание Mouse 4 → речь → текст в поле
- [ ] Отпускание до начала записи не отменяет диктовку
- [ ] Нажатие <250 мс: записи в истории есть, сигнала и уведомления нет
- [ ] Одиночный Esc проходит в приложение; двойной отменяет
- [ ] Тумблер и запасная привязка работают
- [ ] Четвёртое нажатие при полной очереди отклонено **сигналом сразу**
- [ ] Диктовка выключена → Mouse 4 снова «Назад»
- [ ] Снятый хук обнаружен и переустановлен; во время UAC ложной переустановки нет

**Доставка**
- [ ] Текст в буфере **всегда**, независимо от исхода
- [ ] Фокус ушёл → громкий сигнал + уведомление с `Ctrl+Alt+Z`, никуда не вставлено
- [ ] Чужая копия во время диктовки **не затёрта**
- [ ] Вставка в PuTTY и cmd; **переводы строк в терминал заменены пробелами**
- [ ] Поле пароля не заполнено, есть уведомление
- [ ] `Ctrl+Alt+Z` не отдаёт текст из дорожки повторов без подтверждения

**Система**
- [ ] Убийство проводника → значок вернулся
- [ ] Плашка не крадёт фокус
- [ ] Автозапуск виден в Параметрах; отключение не отменяется нами
